# 项目介绍
本项目 (***N.E.K.O_Plugins***) 是 (***N.E.K.O***)[https://github.com/Project-N-E-K-O/N.E.K.O] 项目的的插件市场项目，用于为 N.E.K.O项目提供插件后端支持。

## 项目技术
- 项目采用 Python 3.11 作为主要开发语言。
- 前端采用 React + TypeScript + Radix UI + Tailwind CSS作为主要技术栈。
- 前后端整合，采用 RESTful API 接口。

### 项目技术实现

#### 后端数据存储
- 项目采用 SQLite 数据库作为主要数据存储。
- 项目使用异步多线程技术，提高并发处理能力。
- 项目采用数据库事务机制，确保数据一致性。
- 项目采用数据库索引机制，提高查询效率。

##### 数据库计划
- 实现三层数据库架构（业务适配器层、数据库抽象层、数据库驱动层）
- 数据库抽象层负责数据库操作，数据库驱动层负责与数据库交互。
- 业务适配器层负责处理业务逻辑，与数据库抽象层交互。
- 数据库驱动层负责与数据库交互，执行 SQL 语句。
- 数据库抽象层负责定义数据库操作接口，与业务适配器层交互。
- 业务适配器层集成SQL语句，放开内部调用API用于业务对接。

#### 用户管理
- 项目后端设计用户管理模块，适用于内部管理员审核插件调整设置等操作。
- 利用前端入口设计管理后台，便与管理。
- 项目采用 JWT 认证机制，确保用户身份安全（Token 验证、过期时间、刷新机制等、项目启动自动轮换JWT密钥）。
- 用户采用密码哈希存储，确保用户密码安全。
- 用户采用多段权限划分，实现细致化权限管理。
- 权限列表（每个权限包含多个子权限、以子权限为单位进行权限管理、允许用户自定义权限组合、允许直接划分大权限组自动包含所有子权限）：
    - 系统权限（管理系统设置、SMTP设置、系统日志查看）
    - 插件权限（插件审核、插件管理、插件提交日志查看、AI审核内容管理）
    - AI管理权限（AI相关设置、API URI管理、API密钥管理、AI提示词管理、AI模型设置、AI模型调用日志查看、AI模型调用次数统计、AI审查组调用的模型）
- 权限组：用户自行定义权限组，每个权限组包含多个权限。
- 权限组合：用户可以将多个权限组组合起来，实现权限的自定义管理。
- 权限继承：用户可以将权限组的权限继承给其他权限组，实现权限的继承管理。
- 树权限：用户可以将权限组组织成树状结构，实现权限的层级管理。
- 权限管理：用户可以在管理后台查看、添加、删除、修改权限组、权限组合、权限继承、树权限等操作。
- 超管拥有所有权限，可以对所有用户进行权限管理。
- 管理权限需要系统权限下的“权限管理”和“用户管理”两个权限。

#### AI模块
- 设计AI审核模块，用于处理AI审核相关操作。
- AI审核：
    - 审核插件内容：用户可以对插件提交的内容进行审核，包括插件代码、插件文档、插件元数据等。
    - 审核结果管理：用户可以查看、修改、删除审核结果。
    - 审核完毕后，用户可以对插件进行发布或拒绝操作。
    - AI审核可以同时调用多个AI模型，实现并行审核，提高审核效率，审核结果独立，意为参考多个AI模型的审核结果，而不是依赖单一AI模型的审核结果。
    - AI审核需要处于独立的沙箱环境中，防止AI模型调用过程中出现安全问题。
    - AI审核需要设置超时时间，防止审核过程过长导致系统性能下降。
    - AI审核需要设置最大调用次数，防止审核过程中调用次数超过限制。
    - AI审核相关设置：用户可以设置审核模型、审核提示词、审核超时时间、审核最大调用次数等参数（需要AI管理权限）。
    - AI审核需要记录审核日志，包括审核时间、审核人、审核模型、审核提示词、审核结果等。
    - AI审核需要设置审核日志保留时间，防止审核日志占用过多存储空间。

#### 插件模块
- 利用前端的插件提交入口，用户可以提交插件。
- 插件提交后，需要通过AI审核并由人工确认才能发布。
- 插件发布后，用户可以在插件市场查看、安装、使用插件。
- 插件市场需要提供插件的元数据、插件仓库地址等信息。
- 插件需要在Github上发布，用户可以在插件市场查看、安装、使用插件。
- 插件审核流程：
    1. 用户提交插件。
    2. clone插件仓库。
    3. AI审核插件内容（主要审核代码，如果不过审则通过用户预留的邮箱通知用户）。
    4. 人工审核插件内容（如果不过审则通过用户预留的邮箱通知用户）。
    5. 获取插件代码文件MD5值（以插件仓库地址、插件版本、插件作者为识别标识，每个代码文件都需要计算MD5值）,保存至数据库。
    6. 插件发布。
    7. 插件市场更新插件列表。

##### 插件签名机制
- 利用插件代码文件MD5值，对插件进行签名。
- 插件签名需要保存至数据库，与插件代码文件MD5值保持一致。
- 流程：
    1. N.E.K.O客户端计算插件代码文件MD5值（对端）。
    2. N.E.K.O客户端从服务器API获取公钥（对端、API调用）。
    3. N.E.K.O客户端使用公钥对插件代码文件MD5值进行签名（对端）。
    4. N.E.K.O客户端将签名发送至服务器API（对端、API调用）。
    5. 服务器API解密签名，获取插件标识（插件仓库地址、插件版本、插件作者）。
    6. 服务器API根据插件标识（插件仓库地址、插件版本、插件作者）寻找数据库中对应的MD5记录，验证签名（对端、API调用）。
    7. 如果签名验证通过，则插件签名验证通过，返回插件签名验证通过的响应。
    8. 如果签名验证不通过，则插件签名验证不通过，返回插件签名验证不通过的响应。
    （注：插件签名验证失败，插件将出现安全警告。）
- 密钥管理：
    - 服务器端需要生成公钥对（公钥、私钥）。
    - 服务器端需要将公钥保存至数据库，私钥保存至服务器端本地。
    - 服务器端需要设置密钥过期时间，防止密钥过期导致插件签名验证失败。
    - 统一使用ECDSA 256算法进行签名验证（对端、服务器端）。
